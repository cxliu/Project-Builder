How To Add A Toolchain to M-Build

 

Toolchains

A toolchain will generally consist of one or more of the following tools: library archiver, compiler, linker, and assembler.  The toolchains built into M-Build are

  1. Linux style gcc (compiler and assembler), g++ (compiler), ar (archiver), ld (linker)
  2. Visual studio compiler, linker, and archiver

As of the date this was written, some ARM and Texas Instruments plugins have been created but for other toolchains a plugin will likely be needed.

 

Elements of a toolchain plugin

An M-Build toolchain plugin consists of primarily two types of files:

  1. Manifest files which provide all the information needed to use the toolchain.  These will provide the following information:
    1. Compiler names (e.g. gcc) and flags (e.g. -I include_directory).
    2. Linker name and flags
    3. Archiver name and flags for libraries
    4. Operating system file name extensions (e.g. '.exe'), if needed, for operating systems other than those built into M-Build
    5. Processor names, if needed, for processors other than those built into M-Build
  2. Python Plugin file(s).  This will tell M-Build what configurations of OS, processor type, toolchain, build system (e.g. Visual Studio), and flavour (e.g. debug, release) are valid for this Plugin.

For example your plugin might consist of the following files, which should be placed in a directory named for the plugin.  This guide will follow the example plugin which adds support for Cygwin 'operating system' and tools.  This plugin is very similar to a linux gcc toolchain but this will be a plugin.  It can be found at

\depot\licensing\mbuild_plugins\mbuild_example_plugin

manifest.mb  (This file imports all of the following files)
cygwin.p2_plugin
cygwin_compiler.mb
cygwin_assembler.mb
cygwin_linker.mb
cygwin_archiver.mb
cygwin_operating_systems.mb
cygwin_processors.mb

Keep in mind that cygwin is not an officially supported configuration, it is supplied here only as an example plugin.

 

 

Manifest File Data

Processor Types

You will want to enumerate any new processor types you will want to introduce with your plugin.  We don't need any additional processors for the example plugin, but we have left it in place as an example of how to enumerate them if your plugin needs new processors.

From cygwin_processors.mb:

@doc keyword(processor_example) = 'An example processor'

 

# The following lines will automatically @add the keyword 'processor_example'

[default]

@att VALUES.processor+='example'

This adds a value to the processor dimension, and adds the keyword associated with that processor so that you can use it in your other plugin files to modify compile flags etc.

Operating Systems

The operating systems manifest allows you to tell M-Build about any new operating systems relevant to your toolchain.

From cygwin_operating_systems.mb:

@doc keyword(os_cygwin) = "Cygwin under Windows"

 

# Adding the following line automatically declares the keyword [os_cygwin]

[default]

@att VALUES.os+='cygwin'

 

# Specify the extension for a dynamic library in the cygwin operating system

[os_cygwin.config_type_dynamic_lib]

@att EXT='.so'

 

# Specify the extension for a static library in the cygwin operating system

[os_cygwin.config_type_static_lib]

@att EXT='.a'

 

# Specify the extension for an executable in the cygwin operating system

[os_cygwin.config_type_console]

@att EXT='.exe'

This adds the operating system keyword, and filename extension defaults for the operating system.  The presence of the keyword (os_cygwin) allows compilation flags to be changed based on the presence of the new operating system.

Python Plugin File

The python based plugin file, normally called toolchain.p2_plugin (e.g. cygwin.p2_plugin) will allow your M-Build project files to pull in the new toolchain, as well as define which configurations of dimensions (os, processor, etc.) are valid for the toolchain.

A simple plugin file can look like the example cygwin.p2_plugin shown here:

# Cygwin_support will be available for including in a project file after adding this section

def get_symbols():

    return {'cygwin_support': cygwin_support,

           }

 

# Specify the build tools valid for this plugin (e.g. make or visual studio)

build_tool = ['make']

    

# Specify the operating systems which are valid for this plugin        

operating_systems = ['cygwin']

 

# Specify the processors which are valid for this plugin

cores = ['x86',

         'amd64']

        

# Specify the toolchains which are valid with this plugin           

toolchains = ['cygwin_gnu']

 

# Specify the flavours which you want built for this plugin

flavours = ['release','debug']

 

# Define the list of configurations which works for cygwin

# We are requiring here that the toolchain, processor, operating system, and tool all be defined

# and are restricted to certain values

 

def toolchain_cygwin_is_valid(configuration):

    if not configuration['tool'] == 'make':

        return False

    

    if not configuration['toolchain'] in toolchains:

        return False

    

    if not configuration['processor'] in cores:

        return False

    

    if not configuration['os'] in operating_systems:

        return False

    

    if not configuration['flavour'] in flavours:

        return False    

 

    return True

 

 

# Creates a configuration which allows only combinations from the above function

# If a project file includes cygwin_support, these combinations will be built

 

cygwin_support = BuildConfiguration(['os', 'processor', 'flavour', 'toolchain', 'tool'], toolchain_cygwin_is_valid)

 

 

 

Plugin Build Configuration Calls

Plugin build configuration calls are used to determine what configurations are valid for a toolchain, i.e. to construct a more complex build configuration than the above example did using the 'BuildConfiguration' function.

In order to have a good understanding of the plugin functions you will want to reference the M-Build file 'mbuild.p2_python'.  In particular the following functions allow build configurations to be combined to create the entire configuration you wish to build.  Each of these mbuild_function() will construct a build configuration, usually by taking in a sub-build configuration as an argument. 

A build configuration has two components:

  1. A set of dimensions and a set of values for each dimension.  For example most build configurations will consist of the standard set of dimensions:

standard_dimensions = ['os', 'processor', 'flavour', 'toolchain', 'tool']

  1. An is_valid() function which takes a permutation of dimension values (e.g. 'windows', 'x86', 'release', 'gnu', 'make') and returns whether or not that permutation is valid for the build configuration.  The following is the is_valid function which is built into the mbuild_standard build configuration described below:

def is_valid(config):

# Only use M-Build sanctioned toolchains

if not config['toolchain'] in ['gnu', 'msvs']:

return False

 

# Only use M-Build sanctioned processors

if not config['processor'] in ['x86', 'amd64']:

return False

 

# Only use M-Build sanction operating systems

if not config['os'] in ['windows', 'linux', 'osx']:

return False

 

# Only use M-Build sanctioned tools

if not config['tool'] in ['make', 'msvs']:

return False

 

# Only use M-Build sanctioned flavours

if not config['flavour'] in ['debug', 'release']:

return False

 

# Visual Studio plays with Visual Studio only

if config['toolchain'] == 'msvs':

if config['tool'] != 'msvs':

return False

if config['tool'] == 'msvs':

if config['toolchain'] != 'msvs':

return False

 

# Visual Studio only supports windows

if config['toolchain'] == 'msvs':

if not config['os'] == 'windows':

return False

 

# Windows and OSX support x86 and amd64

if config['os'] in ['windows', 'osx']:

if not config['processor'] in ['x86', 'amd64']:

return False

 

# gcc only targets x86 and amd64

if config['toolchain'] == 'gnu':

if not config['processor'] in ['x86', 'amd64']:

return False

 

return True

mbuild_standard

This call doesn't take any arguments, it simply returns the standard build configuration which M-Build supports through built-in plugins.  The dimensions and is_valid function which this standard build configuration uses is given above.

mytoolchain_buildconfig = mbuild_standard

BuildConfiguration(dimensions, is_valid)

This call allows you to construct a build configuration out of an arbitrary set of dimensions and an is_valid function which will tell M-Build which permutations of these dimensions to build. An example call:

cygwin_support = BuildConfiguration(['os', 'processor', 'flavour', 'toolchain', 'tool'], toolchain_cygwin_exception)

This will create a build configuration using the dimensions specified, and when M-Build is generating a list of values for each of these dimensions by going through a project's makefiles, it will send each permutation of these sets of values to the is_valid function 'toolchain_cygwin_exception', which will return either True or False for each permutation.

mbuild_restrict(build_configuration, match_function)

The first parameter is a BuildConfiguration object, the second is a dictionary mapping dimension names to allowed values. Only builds matching the dictionary will be used. For example:

mbuild_restrict(mbuild_standard, {'os': ['linux', 'osx']})

mbuild_pretend(dimension_name, remap, base_configuration)

Takes a dimension name, a map, and a base BuildConfiguration and modifies the "is_valid" function of the base configuration so it is as if the dimension has had the map applied to it. For example, if there is a BuildConfiguration that forces a dimension 'processor' to be 'x86', along with several other restrictions.

E.g.

mbuild_pretend('os', {'ubuntu': 'linux'}, mbuild_standard)

Here we are using mbuild_pretend to keep the other restrictions from the mbuild_standard is_valid function, while allowing the value 'ubuntu' for operating system as well.

mbuild_union(build_configurations)

This takes multiple BuildConfiguration objects (which must have the same set of dimensions) and combines them together.  If any one of the is_valid functions of all the build configurations returns 'True', the configuration will be valid.

mbuild_conditional(build_configuration, condition)

This allows you add an additional is_valid function to build configuration you're building the chain of checks doneto set an arbitrary python function which takes a single argument and returns True or False.

E.g.

mbuild_conditional(mbuild_standard, personal_is_valid)

In this example both the mbuild_standard is_valid function AND the personal_is_valid function must be satisfied with a 'True' result for the configuration to be permitted.

mbuild_exception(build_configuration, exception)

This function takes a build configuration and adds a special type of is_valid function (passed in as the 'exception' argument) which accepts the result from the is_valid of the base build_configuration as an argument. 

E.g.

def my_exception(permutation, base_config_answer, base_config_is_valid)

if base_config_answer == False

return False

 

if permutation['processor'] == ['x86']

return False

 

return True

 

mbuild_exception(mbuild_standard, my_exception)

In this example the base configuration's answer will be taken, unless the processor is 'x86'.

require(mbuild_function)

The require function is needed in your plugin python file whenever any of the above build configurations will be used.  For example

require('mbuild_standard')

require('mbuild_exception')

arm_support = mbuild_exception(mbuild_standard, toolchain_rvct_exception)

 

 

Makefile Template Plugins

Another plugin file which M-Build uses is a Makefile plugin file.  Every toolchain requires at least a nominal 'make' plugin file.

An example plugin file can be found at \\depot\licensing\mbuild\main\mbuild\data\plugins\gnu_toolchain.t3_plugin

# Export toolchain makefile plugin function name.  Name must start with 'makefn'

def get_symbols():

    return {'makefn_toolchain_gnu': makefn_toolchain_gnu,

           }

 

# These functions and classes can be found in the gnulike_toolchain_template.t3_plugin file

require('t3_gnulike_compile_c')

require('t3_gnulike_compile_cpp')

require('t3_gnulike_compile_asm')

require('t3_gnulike_link')

require('t3_gnulike_archive')

require('t3_GnulikeToolchain')

 

# Return a Toolchain object, in this case the GnulikeToolchain object

def makefn_toolchain_gnu():

    

    # First argument to each of these is the set of suffixes of input files

    # Second argument is the set of suffixes of output files

    # For example a gnu C compiler takes in .c files and outputs .o object files and .d dependency files

    compile_c_action = t3_gnulike_compile_c(['.c'], ['.o','.d'])

    compile_cpp_action = t3_gnulike_compile_cpp(['.cpp'], ['.o','.d'])

    compile_asm_action = t3_gnulike_compile_asm(['.s'], ['.o'])

    # The second argument is omitted for linker and archiver as the output suffix is taking from the manifests

    # for executable and library respectively

    link_action = t3_gnulike_link(['.o','.lib','.a','.so'], [])

    archive_action = t3_gnulike_archive(['.o','.lib','.a','.so'], [])

    

    # function and class names from the gnulike template

    toolchain_class = t3_GnulikeToolchain

 

    # Toolchain definition consists of 3 types of objects

    # Toolchain class: consists of actions

    #      - Action class: e.g. 'Compile'  Each action consists of commands

    #           - Command class e.g. 'gcc hello_world.c'

 

    # This name equal the dimension corresponding to 'toolchain'

    toolchain_name = 'gnu'

    # This tells M-Build to 'include' all files of this suffix which are generated into the makefile

    dependency_extensions = ['.d']

    # The set of all actions, these will be translated into makefile rules by M-Build

    action_defs = [compile_c_action, compile_cpp_action, compile_asm_action, link_action, archive_action]

 

    gnu_toolchain = toolchain_class(toolchain_name, action_defs, dependency_extensions)

 

    return gnu_toolchain

If your toolchain has a syntax exactly like the gnu tools you can simply copy the above plugin file, changing references to 'gnu' to the name of your toolchain.

If your toolchain does not follow the syntax of gnu tools you will likely need to create your own 'action functions'. For example the "Compile C" action is described by the function "t3_gnulike_compile_c" in the above example. That function is shown below:

# Constructs a 'compile action' object for describing what steps are needed to compile a C file for this toolchain

def gnulike_compile_c(insuffixes, outsuffixes):

    

    # This just used to access constants

    action_types = GnulikeActionTypes()

    

    #      - Action class: e.g. 'Compile'  Each action consists of commands

    #           - Command class e.g. 'gcc hello_world.c'

    

    

    # names for these maps are important:

    # 'attrib_' indicates a value to look up, anything after the underscore is unused

    # 'string_' indicates a raw string, anything after the underscore is unused

    # 'defines' get prefixed with the value given

    # 'includes' get prefixed with the value given

    # 'subattrib_' indicates a value to look up but not generate a 'makefile variable' for, instead directly place in the rule

    c_dependencies_command_map   = {\

                               'attrib_exe'           : 'MAKE_CCDEP',       # looking up the compiler executable name for in manifests

                               'subattrib_depflags'   : 'MAKE_CCDEPFLAGS',  # looking up flag attribute in manifests

                               'defines'              : '-D',               # prefix for defines

                               'includes'             : '-I',               # prefix for include directories

                               'subattrib_depflags_t' : 'MAKE_CCDEPFLAGS_TARGET',# looking up another flag attribute in manifests

                               'string_preoutstr'     : '$@ ',              # raw string for formatting command

                               'subattrib_out'        : 'MAKE_CFLAGS_OUTPUT_FILE', # prefix string for output files from manifests

                               'string_endstr'        : '$*.d $< '         # raw string for formatting command

                                   }

    c_dependencies_command_order = ['attrib_exe', 'subattrib_depflags', 'defines', 'includes', 'subattrib_depflags_t', \

'string_preoutstr', 'subattrib_out', 'string_endstr']

 

    c_dep_command = GnulikeActionCommand(c_dependencies_command_map, c_dependencies_command_order)

    

    

    c_compile_command_map   = {\

                               'attrib_exe'        : 'MAKE_CC',       # attribute to look for in manifests

                               'attrib_flags'      : 'MAKE_CFLAGS',  # attribute to look for in manifests

                               'defines'           : '-D',               # prefix for defines

                               'includes'          : '-I',               # prefix for include directories

                               'subattrib_out'     : 'MAKE_CFLAGS_OUTPUT_FILE', # attribute to look for in manifests

                               'string_endstr'     : '$@ $< '         # raw string

                              }

    c_compile_command_order = ['attrib_exe', 'attrib_flags', 'defines', 'includes', 'subattrib_out', 'string_endstr']

 

    c_compile_command = GnulikeActionCommand(c_compile_command_map, c_compile_command_order)

    

    c_in_suffixes = insuffixes

    c_out_suffixes = outsuffixes

    c_commands = [c_dep_command, c_compile_command]

    

    c_action = GnulikeAction(c_in_suffixes,

                      c_out_suffixes,

                      c_commands,

                      action_types.GNULIKE_ACTION_TYPE_COMPILE)

 

    return c_action

This type of function can allow you to create very flexible rule types for your makefiles, while still allowing M-Build to simplify the makefiles with appropriate variables and suffix-based rules.

The example above describes the action "Compile C", which consists of two "Commands": generate dependencies and compile.  Your toolchain may have a different order to the command line, or extra flags needed etc, in this case you would make these changes to the 'command map' and 'command order'.
